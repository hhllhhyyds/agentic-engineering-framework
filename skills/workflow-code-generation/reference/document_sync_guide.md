# 文档同步指南

> 每次 Task 实现完成、Review 通过后，在报告前执行文档同步。
>
> **核心原则**：从修改的代码位置出发，自下而上逐层追溯需要更新的文档。

## 自下而上扫描算法

### 层级 1：修改的代码本身

检查当前 Task 修改的每个文件：

| 变更类型 | 需同步的内容 |
|---------|------------|
| 新增公共接口/类型 | 添加 doc comment（目的、参数、返回值、线程安全等） |
| 修改现有接口签名 | 更新 doc comment |
| 删除废弃接口 | 移除对应 doc comment |
| 实现内部逻辑 | 视复杂度添加模块级注释说明设计思路 |

### 层级 2：修改所在的模块/文件

从修改的文件向上看一层：

- 修改的文件是否有对应的**模块说明文档**（如 `docs/design-docs/<module>/README.md`）？→ 更新
- 修改是否改变了模块的**行为或约束**？→ 更新对应的设计文档
- 修改是否涉及 spec 中描述的**设计决策**？→ 更新 spec.md 中对应章节（如实现细节与设计有出入）

### 层级 3：跨模块影响

检查修改是否波及了模块之外：

- 是否**新增了外部依赖**（库、工具、服务）？→ 更新技术栈文档 / 依赖清单
- 是否修改了**模块间的接口契约**？→ 更新架构文档 / 接口文档
- 是否引入了**新的配置项、错误码、Metrics**？→ 更新对应的配置/错误码/Monitoring 文档
- 是否新增/修改了**构建配置**？→ 更新构建说明

### 层级 4：项目级影响

- 是否**新增了技术选型**？→ 更新架构决策记录（ADR）
- 是否修改了**核心架构原则或项目结构**？→ 更新根目录 README / CLAUDE.md 中的项目描述和目录结构
- 是否**废弃了现有能力**？→ 添加废弃说明，标注迁移路径

## 判断原则

| 场景 | 原则 |
|------|------|
| 不确定文档是否需要更新 | **宁可多改，不可漏改** — 过时的文档比没有文档更危险 |
| 文档与代码不一致 | **文档即真相** — 以代码为准，更新文档使之匹配 |
| 发现 spec 与实现有偏差 | 更新 spec.md，并在 task 报告中说明偏差原因 |
| 找不到对应文档 | 不强制创建，在报告中标注"经检查无对应文档，无需更新" |

## 输出

文档同步完成后，在 task 报告中增加「文档同步」小节：

```markdown
### 文档同步

| 文档 | 操作 | 说明 |
|------|------|------|
| `path/to/doc.md` | 更新 | {变更摘要} |
| `path/to/another.md` | 无需更新 | {判断理由} |
```
