# 任务拆解指南

> 本指南用于将 spec.md 的设计方案拆解为结构化的实施任务清单（tasks.md）。
>
> **前置条件**：调用本指南前，SKILL.md 已确保 spec.md 存在且完整。本指南不负责检查 spec 的存在性或质量，只负责拆解。

## 拆解流程

### 第一步：深入理解 spec

1. 完整读取 spec.md
2. 识别所有设计决策和关键约束
3. 提取可交付产物清单（新文件、修改文件、删除文件）
4. 识别 spec 中明确提出的验收标准
5. **识别跨切面关注点**：错误码、日志格式、Metrics 埋点、配置项等贯穿多个任务的内容

### 第二步：代码调研

> 注：SKILL.md 步骤 1-2 已完成初步代码阅读。本步骤在此基础上**补充调研**任务拆解所需的细节。

调用 `codebase-researcher` subagent 进行调研（可并行启动多个），自行判断需要调研哪些方面。

**本步骤需要获取的信息**：
1. **修改面**：哪些函数/类/接口需要变更
2. **依赖面**：哪些调用方会受影响
3. **构建面**：哪些构建配置需要更新
4. **每个预期任务的 context**：直接修改的文件 + 上游（谁调用/谁传入数据） + 下游（谁消费产出）
5. **变更影响概览**的结构化记录

### 第三步：任务拆解

按以下原则拆解。当原则之间发生冲突时，按**优先级从高到低**裁决：

> **原则优先级**：可编译（原则 1） > 依赖拓扑（原则 3） > 风险前置（原则 6） > 粒度适中（原则 2）
>
> 原则 4（可独立验收）和原则 5（spec 全覆盖）是每个任务的必备属性，不参与优先级排序——它们不会与其他原则冲突，必须始终满足。

#### 原则 1：任务有序、可渐进编译（最高优先级）

- 每个任务完成后，代码库必须能编译通过（允许占位桩）
- 后续任务可以在前序任务的基础上渐进构建
- 如果 A 任务产出的接口被 B 任务使用，A 必须排在 B 前面

**占位桩规范**：当任务需要引入尚未实现的函数/方法时，桩必须满足：
1. 能编译通过（正确的签名和返回类型）
2. 运行时返回明确的错误信息（而非默默成功或崩溃）
3. 注释中标注后续实现任务的编号

```cpp
// Stub: will be implemented in Task N
Status FooBar(const Request& req, Response* resp) {
  return Status::NotSupported("FooBar: stub, see Task #N");
}
```

#### 原则 2：粒度适中

一个任务 = **一个内聚的变更单元**。用以下规则判断拆或合：

**必须拆分（任一命中则拆）**：
- 任务涉及 **3+ 个独立设计决策**（不同的数据结构选型、不同的错误处理策略等）→ 按决策边界拆
- 任务修改 **多个不相关模块**（修改点之间无调用/数据依赖）→ 按模块拆

**必须合并（任一命中则合）**：
- 改动与相邻任务在**同一函数/同一类**中，且单独拿出来没有独立验收价值
- 多处改动是**同一模式的机械重复**（如批量替换函数调用）

**独立成任务（不合并到其他任务）**：
- **跨切面定义**（错误码、日志格式、配置项）→ 独立任务，排在使用方前面
- **集成/测试** → 独立任务

#### 原则 3：依赖关系明确

- 每个任务标注前置依赖（`depends_on` 字段）
- 依赖必须是 DAG（无环）——**每拆出一个新任务，立即检查是否与已有任务构成环**
- 如果两个任务没有依赖关系，标注可并行
- 在任务列表开头提供依赖关系总览图（ASCII art）

#### 原则 4：每个任务可独立验收

每个任务必须有明确的验收标准，至少包含：
- **编译验证**：编译通过（基本要求）
- **行为验证**：具体的功能检查点（如 grep 确认删除、接口一致性检查）
- **不破坏约束**：不引入新的编译警告、不破坏现有测试

**验收标准写法要求**：必须是可机械执行的动作，禁止"确保正确"类描述。

| ✗ 不可操作 | ✓ 可操作 |
|-----------|---------|
| "确保接口正确" | "`grep -r 'OldFunc' src/` 无匹配" |
| "性能满足要求" | "编译通过且无新 warning" |
| "功能正常" | "执行测试命令返回预期值" |

#### 原则 5：spec 全覆盖

- 每个 spec 的设计章节必须有对应的任务覆盖
- **每拆完一个 spec 章节对应的任务，立即更新覆盖映射表**
- 在任务列表末尾提供 spec 章节 → 任务的映射表
- 如果某个 spec 章节无法映射到任务，需说明原因

#### 原则 6：安全变更序——先建后迁后删

变更排序遵循**"先建后迁后删"**三步法，确保每一步都可编译：

1. **建**：创建新接口/实现（与旧代码并存）
2. **迁**：迁移调用方到新接口
3. **删**：删除旧代码

其他排序规则（在不违反三步法的前提下）：
- 高风险/高不确定性的改动排在前面
- 接口定义先于实现（先定契约，再填充）
- 基础设施先于业务逻辑
- 跨切面定义（错误码、配置项）先于使用方

### 第四步：写入 tasks.md 与 dev_plan/

将任务清单写入 `tasks.md`（与 spec.md 同目录），并为每个任务创建独立的 `dev_plan/` 文件。**严格按以下模板生成，不要遗漏任何章节**：

```markdown
# 实施任务清单

> 由 spec.md 生成
> 任务总数: N
> 核心原则: [一句话概括拆解策略，例如"先建后迁后删——先构建新接口，迁移调用方，最后清理旧代码"]

## 依赖关系总览

Task 1 (描述)
  ↓
Task 2 (描述)  ← 依赖 Task 1
  ↓
Task 3 (描述)  ← 被 Task 6, 7 依赖

## 变更影响概览

### 文件变更清单

| 文件 | 操作 | 涉及任务 | 说明 |
|------|------|---------|------|
| `path/to/file.h` | 修改 | Task 1, 3 | 新增接口声明 |

### 受影响接口

| 接口 | 变更类型 | 调用方 | 涉及任务 |
|------|---------|--------|---------|
| `Class::Method()` | 签名变更 | `caller1.cc`, `caller2.cc` | Task 1, 3 |

### 构建系统变更

- `path/to/build_config`：新增 `new.cc`（Task 2）

## 风险与假设

| # | 描述 | 影响任务 | 假设/处理 |
|---|------|---------|----------|
| 1 | [Spec 中不完备或有歧义之处] | Task N | [做了什么假设，或如何处理] |

## 任务列表

### 任务 1: [ ] [简短描述]
- 依赖: 无
- spec 映射: spec 章节 X.Y.Z
- dev_plan: `dev_plan/01-task-name.md`

### 任务 2: [ ] [简短描述]
- 依赖: Task 1
- spec 映射: spec 章节 X.Y.Z
- dev_plan: `dev_plan/02-task-name.md`

## Spec 覆盖映射

| Spec 章节 | 任务 | 说明 |
|-----------|------|------|
| X.1 | Task 1, 2 | ... |
| X.2 | Task 3, 4 | ... |
```

### 第五步：生成独立任务文件（dev_plan/）

在 `tasks.md` 写入完成后，为每个任务在 `dev_plan/` 目录（与 `tasks.md` 同目录）下创建独立的 markdown 文件。

#### 文件命名规范

| 任务类型 | 命名格式 | 示例 |
|---------|---------|------|
| 主任务 | `NN-task-name.md` | `03-implement-core-struct.md` |
| 子任务 | `NN.M-task-name.md` | `03.1-define-inner-types.md` |

编号与 `tasks.md` 中的任务编号保持一致。

#### 文件模板

每个 `dev_plan/` 文件包含当前任务的聚焦上下文，供 AI 在实现阶段按需加载：

```markdown
# Task {N}: {任务标题}

## 目标
{详细说明这个任务要做什么，包括涉及的设计决策和约束}

## 涉及文件
| 文件 | 操作 | 说明 |
|------|------|------|
| `path/to/file.cc` | 修改 | {改动说明} |
| `path/to/new.h` | 新建 | {改动说明} |

## 依赖
{前置任务列表，如：Task 1, Task 2}

## context
- `path/to/file.cc:ClassName::Method()` — 直接修改目标
- `path/to/upstream.cc:FuncA()` — 上游调用方，理解数据如何传入
- `path/to/downstream.cc:FuncB()` — 下游消费方，理解产出如何被使用

## 验收标准
- [ ] 编译通过且无新 warning
- [ ] {其他可机械执行的检查项}

## 子任务
- [ ] N.1: {具体步骤}
- [ ] N.2: {具体步骤}
```

#### 输出要求

- `dev_plan/` 文件与 `tasks.md` 必须一致（编号、依赖、文件列表）
- `dev_plan/` 不重复 `tasks.md` 中的全局信息（依赖总览、Spec 覆盖映射等）——只包含单个任务的聚焦上下文
- 创建完成后，在 `tasks.md` 的文件变更清单中添加 `dev_plan/` 目录的引用

## 质量自检

写入 `tasks.md` 和 `dev_plan/` 后，对照以下 checklist 自检（如果自检不通过，修正后重新写入）：

| # | 检查项 | 判定标准 |
|---|--------|---------|
| 1 | **依赖 DAG 无环** | 画出依赖图，确认无循环 |
| 2 | **每个任务编译可通过** | 逐个检查每个任务引用的类型/函数是否已在前序任务或现有代码中定义/声明（含占位桩） |
| 3 | **每个任务有验收标准** | 验收标准可操作（可 grep / 可编译 / 可运行），无"确保正确"类描述 |
| 4 | **spec 全覆盖** | 映射表无空白行 |
| 5 | **粒度合理** | 无单行任务，无跨 5+ 模块任务 |
| 6 | **变更序安全** | 不存在"先删后建"导致中间态编译失败的情况 |
| 7 | **构建系统同步** | 新文件有对应的构建配置更新步骤 |
| 8 | **无冗余任务** | 每个任务有不可替代的产出 |
| 9 | **跨切面无遗漏** | 错误码/日志/配置等跨切面内容有统一定义任务，且排在使用方前面 |
| 10 | **变更影响概览完整** | 文件变更清单与各任务的文件列表一致，无遗漏 |
| 11 | **context 完整** | 每个任务的 context 包含直接修改文件 + 上游调用方 + 下游消费方，且精确到函数级 |
| 12 | **dev_plan 一致** | `dev_plan/` 文件编号、文件列表、依赖关系与 `tasks.md` 完全一致 |

## 硬性规则

1. **本阶段只生成任务清单**：输出 `tasks.md` 和 `dev_plan/`，不修改其他任何文件，不写代码。生成完任务清单后再进入编码阶段
2. **必须先读代码**：不理解现有代码就拆任务是禁止的
3. **必须引用 spec**：每个任务的 `spec 映射` 字段必须有值
4. **必须可编译**：每个任务完成后代码库必须可编译（占位桩必须符合桩规范）
5. **依赖必须 DAG**：如果发现循环依赖，重新拆分任务直到消除
6. **验收标准必须可操作**：不能是"确保正确"之类的模糊描述
7. **先建后迁后删**：涉及替换旧代码的场景，禁止先删除再创建，必须遵循建→迁→删顺序
8. **歧义必须记录**：拆解过程中遇到 spec 有歧义或需要假设的地方，必须在对应任务中明确标注假设，禁止默默补全
